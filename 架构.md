# 攻防战占领点 Mod 架构文档

## 一、项目概述

一个 Minecraft NeoForge 1.21.1 模组，提供攻防战中的据点和区域管理功能。
玩家可以通过**据点方块**、**管理器物品**、**节点图 GUI** 和 **命令** 四种方式管理
据点和区域的占领关系。支持**多队伍自动占领**（基于 Scoreboard 团队）、**占领进度条 HUD**。

**核心概念**：据点 = (中心坐标, 半径) 定义的区域，方块仅为放置标记。

## 四、核心架构 — 分层数据访问

```
                          ┌─────────────────────────────────┐
                          │       消费者层                    │
                          │  CapturePointBlockEntity         │
                          │  ModCommands / CapturePointGUI   │
                          └──────────┬──────────────────────┘
                                     │ 依赖
                          ┌──────────▼──────────────────────┐
                          │     ICaptureDataAccess 接口      │  ← 统一契约
                          │  (read / write / query / batch)  │
                          └──────────┬──────────────────────┘
                                     │ implements
                          ┌──────────▼──────────────────────┐
                          │   ServerCaptureDataAccess        │  ← 服务端实现
                          │   (单纯委派CaptureManager)       │
                          └──────────┬──────────────────────┘
                                     │ delegates
                          ┌──────────▼──────────────────────┐
                          │   CaptureManager (SavedData)     │  ← 数据持久化层
                          │   points: Map<String, PointEntry> │
                          │   zones : Map<String, ZoneEntry>  │
                          └─────────────────────────────────┘
```

### 设计要点

- **`ICaptureDataAccess`** 定义所有数据操作的统一契约，包含据点 CRUD、区域 CRUD、查询方法、批量快照、版本控制、**队伍/进度操作**
- **`ServerCaptureDataAccess`** 包装 `CaptureManager`，所有操作单纯委派给底层 SavedData
- 消费者不直接依赖 `CaptureManager`，只依赖接口，便于测试和未来替换实现
- 获取方式：`ICaptureDataAccess.server(serverLevel)`

## 五、数据模型

### CapturePointEntry (据点记录)

```java
record CapturePointEntry(
    String name,                    // 唯一名称
    BlockPos pos,                   // 世界坐标（区域中心）
    boolean captured,               // 是否已被占领
    @Nullable String ownerTeam,     // 占领团队的名称 (null=无主)
    @Nullable String capturingTeam, // 正在占领中的团队 (null=无)
    int captureProgress,            // 占领进度 0-100 (100=已完成)
    double radius,                  // 判定半径 (1-100)
    int displayColor,               // 显示颜色 (ARGB)
    boolean showRange               // 是否显示范围轮廓
)
```

**关键常量**：`MAX_PROGRESS = 100` — 进度到达 100 时自动完成占领

**withCaptured() 行为**：设为 true 时自动填充 ownerTeam/capturingTeam/captureProgress；设为 false 时清空。

### ZoneEntry (区域记录)

```java
record ZoneEntry(
    String name,                  // 唯一名称
    List<String> capturePoints,   // 包含的据点名称列表
    @Nullable String requiredZone // 前置依赖区域 (攻防模式)
)
```

### 业务规则

- **区域完全包含据点** — 据点通过 `capturePoints` 列表归属区域
- **自动占领** — 玩家进入据点范围（基于 Scoreboard 团队），进度自动推进，约 5 秒完成
- **多队伍竞争** — 统计范围内各队伍人数，取人数最多的队伍推进进度
- **区域占领判定** — 区域内所有据点被占领时判定为已占领
- **攻防模式依赖** — 区域有 `requiredZone` 时，前置区域未占领则此区域不可访问
- **方块仅存绑定名** — `CapturePointBlockEntity` 只持久化 `boundPointName`

## 六、服务端占领逻辑 (ServerTickHandler)

```
ServerTickEvent.Post (每 tick)
  ├─ tickCounter % 10 == 0 → processCaptureLogic(level)
  └─ tickCounter % 40 == 0 → syncAllBoundBlocks (安全网)

processCaptureLogic(ServerLevel):
  for each entry in getPoints():
    if !entry.showRange() → continue (未开启范围显示的据点不自动占领)
    检测 entry.pos() 半径 entry.radius() 内的所有玩家
    统计各 Scoreboard 团队人数，取人数最多的队伍
    根据队伍关系处理：
      ├─ 无玩家 → 清除 capturingTeam
      ├─ 已被此队伍占领 (ownerTeam == dominantTeam) → 保持进度满
      ├─ 多队伍混合 → 清除 capturingTeam (等待单队伍独占)
      └─ 单队伍 → 每次 +2% 进度，设置 capturingTeam
                 进度达 100 → captured=true, ownerTeam=该队伍
```

### 占领判定流程

```
玩家站在据点范围内
  ├─ 无队伍 → 忽略（必须加入 Scoreboard 团队）
  ├─ 已占领且是己方队伍 → 保持 100%
  ├─ 已占领且是敌方队伍 → 不能占领（需其他队友不在时）
  ├─ 正在被己方占领 → 继续推进进度
  ├─ 正在被敌方占领 → 需要范围内无敌人时转为自己占领
  └─ 空闲 (capturingTeam=null) → 开始占领，设置 capturingTeam
```

## 七、客户端/服务端分离

```
┌──────────────────────────────────────────────────────────┐
│  客户端 (Client)                                         │
│  ┌─────────────────┐  ┌──────────────┐  ┌────────────┐  │
│  │ CapturePointGUI  │  │ BlockEntity  │  │ ManagerItem│  │
│  │ (节点图编辑器)    │  │ (7功能菜单)   │  │ (打开GUI)   │  │
│  └────────┬────────┘  └──────┬───────┘  └────────────┘  │
│           │                  │                            │
│           │             ┌────▼────┐                       │
│           │             │进度条HUD │ ← CaptureProgressOverlay│
│           │             └─────────┘                       │
│           │  网络包          │  网络包                     │
│           │  (BlockEntity    │  (BlockEntity              │
│           │   ActionPayload) │   ActionPayload)           │
│           │                  │                            │
│           │  客户端缓存       │  ClientCaptureDataCache     │
│           │  ← 服务端→客户端  │  (CaptureDataSyncPayload)   │
│           │  (含 ownerTeam,   │  capturingTeam, progress)  │
└───────────┼──────────────────┼────────────────────────────┘
            │                  │
┌───────────▼──────────────────▼────────────────────────────┐
│  服务端 (Server)                                           │
│  ┌──────────────────────────────────────────────────┐     │
│  │  ServerTickHandler.processCaptureLogic()          │     │
│  │  → ICaptureDataAccess → CaptureManager           │     │
│  │  → 每 10 tick 检测玩家距离 + 更新进度            │     │
│  └──────────────────────┬───────────────────────────┘     │
│                         │                                  │
│  ┌──────────────────────▼───────────────────────────┐     │
│  │  CaptureManager (SavedData, 自动持久化到磁盘)      │     │
│  │  数据变更后: syncAllBoundBlocks + broadcastToAll   │     │
│  └──────────────────────────────────────────────────┘     │
│                                                            │
│  ModCommands 也通过 ICaptureDataAccess 直接访问 (服务端)   │
│  PlayerLoggedInEvent → 发送初始数据到客户端                │
│  ServerTickHandler (NeoForge.EVENT_BUS 手动注册)           │
└──────────────────────────────────────────────────────────┘
```

### 原则

- **客户端永不直接写 CaptureManager** — 所有写入通过网络包发送到服务端
- **读取** — 客户端通过 `ClientCaptureDataCache` 读取同步自 CaptureManager 的数据
- **渲染** — `CapturePointWorldRenderer` 从 `ClientCaptureDataCache` 读取据点数据，在世界坐标系中画圆
- **HUD** — `CaptureProgressOverlay` 从 `ClientCaptureDataCache` 读取队伍/进度数据，渲染进度条
- **同步网络包** — `CaptureDataSyncPayload` 服务端→客户端，版本去重，仅数据变更时发送

## 八、网络通信

### BlockEntityActionPayload (客户端→服务端)

```java
record BlockEntityActionPayload(BlockPos blockPos, String action, String data)
```

支持动作：
| action | data | 说明 |
|--------|------|------|
| `create_point_at` | `name,x,y,z,radius` | 创建据点 + 绑定方块 |
| `set_radius` | `name,radius` | 设置半径 |
| `set_color` | `name,color` | 设置颜色 |
| `toggle_range` | `name,true/false` | 切换范围显示 |
| `set_captured` | `name,true/false` | 设置占领状态 |
| `zone_set_captured` | `zoneName,true/false` | 设置区域占领状态 |
| `add_to_zone` | `zoneName,pointName` | 据点加入区域 |
| `remove_from_zone` | `pointName` | 据点移出区域 |
| `block_unbind` | `pointName` | 方块解除据点绑定 |

### CaptureDataSyncPayload (服务端→客户端)

```java
record CaptureDataSyncPayload(Map<String, PointRenderData> points)

record PointRenderData(
    BlockPos pos,
    double radius,
    int displayColor,
    boolean showRange,
    boolean captured,
    @Nullable String ownerTeam,
    @Nullable String capturingTeam,
    int captureProgress
)
```

同步触发时机：
- **玩家加入世界** — `PlayerLoggedInEvent` 发送完整数据
- **GUI 保存** — `saveGraph()` 后 `syncAllBoundBlocks` + `broadcastToAll`
- **方块操作** — `handleOnServer` 末尾 `broadcastToAll`
- **周期安全网** — 每 40 tick `syncAllBoundBlocks`（版本去重，数据无变更不发送）

## 九、节点图系统 (LDLib2 nodegraphtookit)

### 架构

```
CapturePointGraph (extends Graph)
  └─ createGraphModel() → CapturePointGraphModel (extends CustomGraphModelImpl)
       ├─ createWire()    — 端口方向修正 + 兼容性检查
       └─ canAssignTo()   — 信号类型 TypeHandle 严格匹配

CapturePointGraphView (extends GraphView)
  ├─ 隐藏默认面板/标题栏/物品库
  ├─ 右键菜单 → CapturePointGraphDialogs 对话框 + 删除连线
  ├─ 定时刷新 → CapturePointGraphScreen.refreshNodeTitles()
  └─ deleteSelectedElements() — 直接删除选中连线（WireModel）

CapturePointGraphScreen
  ├─ open()        — 创建全屏 UI, 加载数据, 设置回调
  ├─ saveGraph()   — 构建快照 → 冲突检测 → applyGraphSnapshot → syncAllBoundBlocks
  └─ buildSnapshotFromGraph() — 解析节点+连线 → 数据映射
```

### 节点类型

| 节点 | 端口 | 选项 | 端口容量 |
|------|------|------|----------|
| **CapturePointNode** (绿色) | 输出: `point_signal` (POINT_SIGNAL) | captured, position, radius, display_color, show_range | SINGLE |
| **CaptureZoneNode** (蓝色) | 输入: `point_in` (POINT_SIGNAL), 输出: `zone_out` (ZONE_SIGNAL), 输入: `required_zone` (ZONE_SIGNAL) | captured, required_zone, points, description, edit_points | **point_in = MULTIPLE** |

### 连线类型

| 连线 | 含义 | 保存时的处理 |
|------|------|-------------|
| `point_signal` → `point_in` (MULTIPLE) | 据点属于该区域 | 从 WireModel 解析，**唯一数据源**（不依赖 edit_points） |
| `zone_out` → `required_zone` | 区域依赖关系 | 从 WireModel 解析，作为 requiredZone 来源 |

### 保存流程

```
用户点击 Save
  ↓
saveGraph()
  ↓
buildSnapshotFromGraph()
  1. 遍历所有 GraphElement → 分类为 PointModel / ZoneModel
  2. 遍历所有 WireModel → 解析连线关系 (wireBasedZonePoints / wireBasedRequiredZone)
  3. 读取 PointModel 选项 → 构建 CapturePointEntry (含 captured, ownerTeam=null 待自动占领)
  4. 读取 ZoneModel 选项 + wireBasedZonePoints → 构建 ZoneEntry
  5. 双向同步: 区域captured → 子据点同步; 子据点 → 区域重新计算
  ↓
版本冲突检测 (editMode && snapshotVersion != mgr.getVersion())
  ↓
applyGraphSnapshot(newPoints, newZones) → 原子写入 CaptureManager
  ↓
syncAllBoundBlocks(serverLevel) → 方块实体同步
  ↓
broadcastToAll (CaptureDataSyncPayload) → 客户端缓存更新
```

**关键设计**：区域包含的据点列表**只从连线(WireModel)解析**，不依赖 `edit_points` 选项。
删除连线即删除据点归属关系。`edit_points` 仅为编辑模式的文本展示。

### 编辑模式

- 点击 `[EDIT]` 进入编辑模式 → 面板层出现，选项可编辑
- 右键选中节点 → "编辑属性" 修改所有字段
- 拖拽接线 → `point_in` 端口 **MULTIPLE** 容量，支持多个据点连线到同一区域
- 选中连线 → 右键 "删除连线" 或按 Delete 键直接删除

## 十一、方块系统

### CapturePointBlock

- 继承 `BaseEntityBlock`，渲染模式 `RenderShape.MODEL`
- 右键触发 `CapturePointBlockEntity.openUI()`（客户端打开菜单）

### CapturePointBlockEntity

- **7 个功能按钮**：创建据点、绑定区域、查看状态、设定半径、切换范围、移除绑定、设置占领者
- **方块属性（唯一持久化）**：`boundPointName` — 绑定到 NBT `saveAdditional/loadAdditional`
- **渲染缓存**（不持久化）：`radius`, `displayColor`, `showRange` — 从 CaptureManager 同步
- **读取来源**：方块菜单优先从 `ClientCaptureDataCache` 读取，回退到本地缓存字段
- **写入路径**：所有操作通过 `sendAction()` → `BlockEntityActionPayload` 网络包 → 服务端 CaptureManager
- **没有乐观更新**：所有写入只发网络包，不直接修改本地缓存

## 十三、渲染系统

### CapturePointBlockEntityRenderer

- 留空渲染器 — 方块模型由 `RenderShape.MODEL` 处理
- 圆圈范围渲染已迁移至 `CapturePointWorldRenderer`

### CapturePointWorldRenderer

- 使用 `RenderLevelStageEvent.AFTER_BLOCK_ENTITIES` 事件钩子
- 从 `ClientCaptureDataCache` 读取据点渲染数据（来自服务端网络同步）
- **以据点的实际中心坐标绘制**，而非方块位置
- 32 段圆环 (水平) + 4 条垂直辅助标记
- 视锥体裁剪 — 仅渲染视野内的据点
- 单机和联机均可用（数据源自 `CaptureDataSyncPayload` 网络包）

### ClientCaptureDataCache

- 客户端单例缓存，存储所有据点的渲染数据（含队伍/进度字段）
- 数据来源：`CaptureDataSyncPayload.handleOnClient()`
- 线程安全（synchronized 读写）

### CaptureProgressOverlay

- 监听 `RenderGuiEvent.Pre` 事件
- 从 `ClientCaptureDataCache` 读取队伍/进度数据
- 渲染进度条和状态文字（见第 十 节）
- 仅玩家在据点范围内时显示

## 十五、版本去重与性能

`CaptureDataSyncPayload.broadcastToAll()` 内置版本去重：

```java
// CaptureManager 有版本号，每次数据变更递增
long currentVersion = access.getVersion();
Long lastVersion = LAST_BROADCAST_VERSION.get(dimKey);
if (lastVersion != null && lastVersion == currentVersion) return;
```

- 数据未变更时 **不发送** 网络包
- 单据点 ≈ 100 字节，50 个据点 ≈ 5KB（含队伍/进度字段）
- 玩家操作后即时推送，周期安全网去重后几乎 0 开销
- 占领进度 tick 处理间隔 10 tick (0.5 秒)，性能开销低
